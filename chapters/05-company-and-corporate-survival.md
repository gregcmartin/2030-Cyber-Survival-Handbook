# Company and Corporate Network Survival

[Home](../README.md) | [Previous](04-financial-and-wealth-survival.md) | [Next: Technology Dependence](06-technology-dependence.md)

---

By 2030, companies were not hacked once.

They were continuously negotiated with by hostile automation.

Every employee became a possible entry point. Every vendor became a possible bridge. Every API became a possible argument. Every admin panel became a jackpot.

---

## Inventory Everything

You cannot defend what you cannot name.

Maintain current lists of:

- Domains
- Cloud accounts
- Admin users
- SaaS tools
- Vendors
- APIs
- Service accounts
- Code repositories
- Production systems
- Employee devices
- Critical data stores

Unknown assets are bot food.

---

## Lock Down Identity First

Identity became the battlefield.

Require:

- Strong multi-factor authentication
- Hardware keys for administrators
- Single sign-on where practical
- Least privilege access
- Fast employee offboarding
- Regular access reviews
- No shared admin accounts
- Protected break-glass accounts

Most companies did not lose because their firewall failed.

They lost because the wrong account still had power.

---

## Segment the Network

Flat networks died fast.

Separate:

- Employee devices
- Production systems
- Finance systems
- Backups
- Development environments
- Customer data
- Security tooling

An attacker who reaches one area should not inherit the whole company.

---

## Protect Backups Like Treasure

Backups are not backups if attackers can erase, encrypt, or poison them.

Use:

- Offline or immutable backups
- Regular restore tests
- Separate backup credentials
- Monitoring for backup deletion
- Documented recovery steps

A backup you have never restored is a belief, not a plan.

---

## Prepare for AI Social Engineering

Train employees for attacks that sound perfect.

Attackers will know:

- Org charts
- Writing styles
- Vendor names
- Current projects
- Travel schedules
- Recent meetings
- Personal pressure points

Create rules that do not depend on "seems legitimate."

For example:

- No payment changes without secondary verification.
- No new vendor banking details without finance approval.
- No production access through chat requests.
- No emergency exception without a logged human owner.

---

## Run Defensive AI, But Do Not Worship It

AI defense matters.

Use it to:

- Watch logs
- Detect abnormal behavior
- Summarize alerts
- Triage incidents
- Review code
- Find exposed secrets
- Monitor public attack surface

But keep humans accountable.

AI can help defend the wall.

It should not own the keys to the kingdom without oversight.

---

## Make an Incident Room Before You Need One

Create a written incident plan with:

- Decision makers
- Legal contacts
- Security contacts
- Communications owner
- Customer notification process
- Law enforcement contact path
- Backup communication channels
- Authority to shut systems down

When the attack starts, confusion is damage.
